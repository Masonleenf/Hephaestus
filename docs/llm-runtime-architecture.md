# LLM Runtime Architecture

The package treats `AGENTS.md` as the canonical behavior contract. The three
core meta-agent team members generate or repair runtime adapters as thin
mappings over that core.

## Runtime Matrix

| Runtime | Entry Point | Adapter Files | Memory Access | Verification |
| --- | --- | --- | --- | --- |
| Codex | plugin skill or root `AGENTS.md` | `codex/plugins/agentlas-core-engine-meta-agent/` | Reads repo files and `.agentlas/` contracts | plugin validation plus package verification |
| Claude Code | `/meta-agent` command | `.claude/commands/`, `.claude/agents/`, `.claude/skills/` | Reads root docs and `.agentlas/` contracts | package verification |
| Gemini CLI | `GEMINI.md` | `GEMINI.md`, `.gemini/GEMINI.md` | Reads root docs and `.agentlas/` contracts | package verification |
| Cursor | root rules through `AGENTS.md`-compatible behavior | optional `.cursor/` generated by target repo | Depends on project setup | package verification |
| Generic | `AGENTS.md` | root `AGENTS.md` | Reads files in local project | package verification |

## Rules

- Keep behavior in `AGENTS.md`.
- Keep runtime adapters short and explicit.
- Name unsupported capabilities instead of hiding them.
- Do not duplicate private or hosted-only Agentlas behavior in public adapters.
- Every adapter must point back to the canonical core and the package verifier.

## Model And Tool Selection

The meta-agent should choose models and tools by task:

- architecture and planning: strongest reasoning model available;
- file editing: runtime with filesystem tools and patch support;
- browser or UI validation: runtime with browser automation;
- memory review: runtime that can read the `.agentlas/` contracts and current
  repo files;
- public release: runtime with shell access for verification scripts.
